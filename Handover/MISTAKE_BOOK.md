更新时间：2026-03-05
负责人：Codex（接手开发）

## 1) 关键案例：MathML 复制到 Word 失败（已修复）

### 现象
- LaTeX 复制成功（如 `$$\Delta E = h\nu$$`）。
- MathML 复制后在 Word 中显示为普通文本（如 `ΔE=hν`），不能变成公式块。

### 根因（最终确认）
- 问题核心不是 LaTeX 提取，而是“剪贴板载荷形态 + 目标应用实际采用的通道”。
- 当复制链路被目标应用按文本通道消费时，MathML 会被当普通文本。
- 仅“复制成功”日志不代表“Word 已按公式对象接收”。

### 可用结果特征（对齐参考插件）
- 最终 MathML 结构中存在：
  - `mml:math`
  - `mml:mrow`
  - 首元素包裹 `mml:mpadded lspace="0"`
- 该结构是本项目当前已验证可用的工程基线。

### 本次工程经验
- 必须盯“结果态证据”，不能只盯“API 调用成功”。
- 复制问题优先看三件事：
  - 最终字符串结构（是否含 `mpadded lspace="0"`）。
  - HTML 片段结构（是否被浏览器当成真实节点而非文本）。
  - 目标应用粘贴后的实际行为（Word 是否生成公式对象）。

## 2) 我反复犯错点（必须避免）

1. 过度依赖 fallback 推理，忽略“主路径可验证结果”  
- 错误：在 fallback 分支反复微调，导致用户体验恶化。  
- 纠正：先锁定主路径输出与参考结果逐字对齐，再谈优雅降级。

2. “复制成功”与“功能成功”混淆  
- 错误：把 `clipboard-write-success` 视为功能完成。  
- 纠正：定义成功标准为“Word 粘贴后进入公式对象”。

3. 日志设计不够结果导向  
- 错误：日志只有流程节点，缺少最终 payload 快照。  
- 纠正：增加 `word-mathml-final-snapshot`、`word-mathml-html-snapshot`、`hasWordBlockCompat`。

4. 噪音日志干扰主问题判断  
- 错误：RepairEngine 大量 mutation 日志掩盖 FormulaCopy 信号。  
- 纠正：要求验证时只收集 FormulaCopy 关键日志。

5. 没有第一时间把“参考实现”当作契约  
- 错误：先做理论优化，再回头对齐参考。  
- 纠正：用户已给可用结构时，先“对齐可用结果”，后做抽象。

## 3) 项目内其他高频失误（历史复盘）

1. `chrome.storage.local` 与 `sync` 混用导致配置不同步  
- 表现：开关状态不一致、重启后行为漂移。  
- 对策：明确单一来源，监听同一存储域的变更。

2. `data-math="block"` 哨兵误判为真实 LaTeX  
- 表现：复制出 `$$block$$`。  
- 对策：必须从 annotation 提取真实公式，哨兵值直接判无效。

3. 修复引擎与 DOM 变更的自激回路  
- 表现：页面闪烁、频繁重排。  
- 对策：自变更隔离标记 + 节流 + 只处理目标容器。

4. 功能开关“关不掉”  
- 表现：UI 已关闭但行为仍生效。  
- 对策：初始化、监听、销毁三段生命周期必须一致。

5. 版本错觉（加载了旧 dist）  
- 表现：用户反馈“无变化”，实际跑的是旧 chunk。  
- 对策：每次构建更新 Build Mark，并在日志里打印构建标识。

## 4) 防再犯清单（强制执行）

1. 先定义“用户可见成功标准”，再写代码。  
2. 每次修复必须有“最终结果快照日志”。  
3. 先对齐参考可用结果，再做结构优化。  
4. 每次构建必须更新 Build Mark。  
5. 验证时先收集主链路日志，避免噪音。  
6. 如果用户给了可用样本，优先做字面结构对照。  
7. 不用“我认为”，只用“日志 + 结果 + 对照”。  

## 5) 这次修复的最低验收标准（以后沿用）

1. 控制台出现 `hasWordBlockCompat: true`。  
2. `word-mathml-final-snapshot` 中可见 `mpadded lspace="0"`。  
3. Word 粘贴后是可编辑公式块，而非普通文本。  
4. 普通输入框粘贴行为不被破坏。
