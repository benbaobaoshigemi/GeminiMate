## 一、架构原则

### 1.1 当前架构优势

GeminiMate 采用 **Feature-as-Module** 架构，每个功能是一个独立模块：

```
src/
├── core/                    # 核心基础设施
│   ├── services/            # LoggerService, RepairEngine
│   └── types/               # 公共类型定义
├── features/                # 【新增】每个功能一个文件夹
│   ├── watermarkRemover/    #   ├── index.ts (入口, start/destroy)
│   ├── formulaCopy/         #   ├── service.ts (核心逻辑)
│   ├── mermaid/             #   └── types.ts (类型)
│   ├── quoteReply/
│   ├── timeline/
│   ├── folder/
│   ├── chatWidth/
│   ├── editInputWidth/
│   ├── sidebarWidth/
│   ├── inputCollapse/
│   ├── sendBehavior/
│   ├── titleUpdater/
│   └── wordExport/          # 【独创】Word 导出
├── pages/
│   ├── popup/               # React Popup UI
│   ├── content/             # Content Script 入口 (调度所有 features)
│   └── background/          # Service Worker
└── utils/                   # i18n, DOM helpers
```

**关键设计决策**：
- **UI 与逻辑严格分离**：所有 feature 模块只暴露 `start()` 和 `destroy()` 生命周期方法，以及通过 `chrome.storage` 读写配置。Popup UI 只需渲染开关并写 storage，不需知道具体实现。
- **这意味着 UI 可以被任意替换**——其他程序员可以完全重写 Popup，只要 storage key 不变，功能层完全不受影响。

### 1.2 模块生命周期规范

每个 feature 必须遵循：

```typescript
// features/xxx/index.ts
export function startXxx(): () => void {
  // 1. 从 chrome.storage 读初始配置
  // 2. 注册 MutationObserver / 事件监听
  // 3. 执行首次处理
  // 4. 监听 storage.onChanged 响应运行时配置变更
  // 5. return cleanup 函数 (断开 observer, 移除事件)
}
```

- **文件**: `src/core/services/RepairEngine.ts`
- **三板斧**:
  1. **粗体水合**: `**text**` → `<b>` (橙色虚线 `#ffab40`)
  2. **精确修剪**: `<b> text </b>` → `<b>text</b>` (蓝色虚线 `#40c4ff`)
  3. **边界修复**: `文字$$` → `文字 $$` (绿色虚线 `#b2ff59`)

#### 2.1.3 Mermaid 图表渲染 🔨 待移植

- **Voyager 源码**: `content/mermaid/index.ts` (785行, 单文件)
- **原理**: 
  1. **懒加载**: Mermaid 库约 1MB，只在检测到 mermaid 代码块时才 `import('mermaid')`
  2. **语言检测**: 优先看 Gemini 给出的 `code-block-decoration` 标签。如果标签是 "mermaid" 则直接渲染；如果是 "python" 等具体语言则跳过；如果是 "代码段" 等泛化标签则做内容检测
  3. **内容检测**: 检查代码是否以 `graph`/`sequenceDiagram`/`classDiagram` 等 30+ 关键词开头，且至少 3 行、50 字符以上
  4. **渲染**: 调用 `mermaid.render()` 生成 SVG
  5. **UI**: 创建 `📊 Diagram / </> Code` 双模式切换 + 点击全屏查看器(缩放/拖拽/ESC)
  6. **白空格清洗**: 替换 NBSP `\u00A0`、全角空格 `\u3000`、零宽字符等

- **移植难度**: ⭐⭐⭐ (中等)
  - 需要安装 `mermaid` 依赖
  - 需要处理 Vite 动态 import 打包
  - 全屏查看器可复用 Voyager 代码

- **依赖**: `mermaid` (~1MB, 懒加载)

#### 2.3.3 公式点击复制 🔨 待移植

- **Voyager 源码**: `features/formulaCopy/` (3文件, FormulaCopyService.ts 666行)
- **原理**:
  1. 全局事件委托: 在 `document` 上监听 `click`・`mouseover`・`mouseout`
  2. 节点查找: 判断点击目标是否在 `.katex`/`.math-block`/`.math-inline` 内
  3. LaTeX 提取: 优先从 `<annotation encoding="application/x-tex">` 取，回退到 `data-math` 属性
  4. 多格式输出:
     - **LaTeX**: 直接复制原始字符串，用 `$...$` 或 `$$...$$` 包裹

- **视觉反馈**: 点击时显示蓝色描边 + "Copied!" tooltip (2秒自动消失)
- **移植难度**: ⭐⭐⭐
- **依赖**: `temml` (已安装)

#### 2.3.4 水印移除 (NanoBanana) 🔨 待移植

- **Voyager 源码**: `content/watermarkRemover/` (7文件 + fetchInterceptor.js)
- **原始算法**: MIT 许可证，来自 `gemini-watermark-remover` by journey-ad
- **原理 (逆向 Alpha Blending)**:
  1. Gemini 加水印公式: `watermarked = α × logo + (1-α) × original`
  2. 逆向求解: `original = (watermarked - α × logo) / (1-α)`
  3. 预置两张参考图 `bg_48.png` / `bg_96.png`（对纯白背景的水印截图）
  4. 从参考图计算 alpha map (`max(R,G,B)/255` per pixel)
  5. 根据图片尺寸自动判断水印大小 (>1024px 用 96px, 否则 48px)
  6. 水印位置: 右下角，margin 64px 或 32px
- **双层架构**:
  - **Content Script (ISOLATED world)**: 处理预览图的水印移除 + 与 Background Script 通信获取 CORS-free 图片
  - **fetchInterceptor.js (MAIN world)**: 拦截下载请求，修改 URL 参数获取原图，通过 DOM bridge (`data-request`/`data-response`) 将图片传给 Content Script 处理
- **UI**: 在下载按钮旁显示 🍌 指示器 + 状态 toast (下载中/处理中/成功/失败)
- **所需权限**: `host_permissions` 中需要 `*.googleusercontent.com`
- **移植难度**: ⭐⭐⭐
- **资产依赖**: `bg_48.png`, `bg_96.png` (水印参考图)

### 2.4 重磅独创: Word 文档导出 🆕

#### 2.4.1 技术方案

**管线设计**:

```
Gemini 对话 DOM
      │
      ▼
  ① 提取原始文本 (HTML → Markdown)
      │
      ▼
  ② LaTeX 修复 (我们的 RepairEngine)
      │
      ▼
  ③ LaTeX → MathML (temml 库)
      │
      ▼
  ④ MathML → OMML (mathml2omml 库)
      │
      ▼
  ⑤ 组装 DOCX (docx 库)
      │
      ▼
  ⑥ 下载 .docx 文件
```

#### 2.4.2 各阶段详解

**① 提取原始文本**
- 遍历 `.message-content` 下的 DOM
- 识别 `<p>`, `<h1>`~`<h6>`, `<ul>/<ol>`, `<table>`, `<code-block>`, `<math-inline>/<math-block>`
- 将结构化内容序列化为中间表示 (IR)

**② LaTeX 修复**
- 复用 `RepairEngine` 中的 `sanitizeMathContent()` 和 `revertNativeMath()` 逻辑
- 确保所有公式都恢复为纯 LaTeX 字符串

**③ LaTeX → MathML**
- 使用 `temml` (已安装) 将 LaTeX 转为 MathML XML
- `temml.renderToString(latex, {displayMode})` → `<math xmlns="...">...</math>`

**④ MathML → OMML (Office Math ML)**
- Word 内部不使用 MathML，使用微软自研的 OMML 格式
- 库选择: `mathml2omml` (MIT, 轻量) 或 `@seewo-doc/docx-math-converter` (更完整)
- 转换: `mathml2omml(mathmlString)` → OMML XML 字符串

**⑤ 组装 DOCX**
- 使用 `docx` 库 (npm, MIT 许可)
- 映射:
  | 中间表示 | docx API |
  |----------|----------|
  | 标题 | `new Paragraph({ heading: HeadingLevel.HEADING_1 })` |
  | 段落 | `new Paragraph({ children: [new TextRun("...")] })` |
  | 粗体 | `new TextRun({ bold: true, text: "..." })` |
  | 代码块 | `new Paragraph({ style: "code" })` |
  | 公式(行内) | `new Math({ children: [new MathRun(ommlXml)] })` |
  | 公式(块级) | `new Paragraph({ children: [new Math(...)] })` |
  | 表格 | `new Table({ rows: [...] })` |
  | 列表 | `new Paragraph({ bullet: { level: 0 } })` |

**⑥ 下载**
- `Packer.toBlob(document)` → `Blob`
- `URL.createObjectURL(blob)` → `<a download="对话标题.docx">`

#### 2.4.3 新增依赖

| 包名 | 大小 | 用途 |
|------|------|------|
| `docx` | ~150KB | 生成 .docx 文件 |
| `mathml2omml` | ~20KB | MathML → OMML 转换 |
| `temml` | ~50KB | LaTeX → MathML (已安装) |

#### 2.4.4 预期效果
- 一键将当前对话导出为学术级 Word 文档
- 公式在 Word 中显示为原生可编辑公式（不是图片！）
- 代码块有语法高亮样式
- 表格完整保留

## 六、关键技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| Gemini DOM 结构变更 | 所有 DOM 操作功能失效 | 使用多层选择器兜底 (Voyager 的做法) |
| Chrome CSP 限制 | fetchInterceptor 无法注入 MAIN world | 使用 `chrome.scripting.executeScript` 注入 |
| mermaid 包体积 (~1MB) | 首屏加载变慢 | Voyager 已验证的懒加载方案 |
| OMML 公式兼容性 | Word 渲染不正确 | 需要在多版本 Word 中测试 |
| GPL v3 传染 | 必须开源 | 项目本身计划开源，无影响 |
